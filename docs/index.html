Overview

System Overview

HoloAssist-AI is the AI extension to the RS2 HoloAssist project, built for the 41118 Artificial Intelligence in Robotics course. An Intel RealSense D435i (simulated in Gazebo Ignition Fortress) captures RGB-D point clouds. Guy Smith’s DBSCAN pipeline extracts 3D cube centroids with ~1.6 cm accuracy. John Chen’s staged MuJoCo sub-policies and Sebastian Baudille’s Isaac Lab grab policy (4096 GPU-parallel envs) train a PPO agent to control a UR3e + RG2 arm. Nic Sabatini built the original hand-eye calibration and perception system (Park solver, <2 mm accuracy) that anchors the coordinate frames.

Quick Start

Quick Start

Clone, run setup, and launch the simulation:

# Build the ROS workspace (one-time) git clone https://github.com/Sebastian-Baudille/HoloAssist-AI.git cd HoloAssist-AI source /opt/ros/humble/setup.bash cd ros2_ws && colcon build --packages-select holoassist_sim --symlink-install source install/setup.bash && cd .. # Terminal 1 — Gazebo + RViz (leave running) source /opt/ros/humble/setup.bash && source ros2_ws/install/setup.bash ros2 launch holoassist_sim sim.launch.py # Terminal 2 — Capture a point cloud source /opt/ros/humble/setup.bash && source ros2_ws/install/setup.bash ros2 run holoassist_sim save_pointcloud.py \ --params ros2_ws/src/holoassist_sim/config/sim_params.yaml # Terminal 3 — Run DBSCAN clustering (no venv needed if open3d/sklearn installed) python3 clustering/detect_cubes.py --no-viz

See LAUNCH.md for the full command reference including RL training and parallel workers.

Open Tasks

Remaining work & known gaps

Items that are not yet done at project end. Ordered roughly by dependency (earlier items unblock later ones).

Perception & clustering

  • Add validation graphs to the clustering pipeline ⚠
    Validation was done numerically via verify_detection.py (Hungarian matching, 60 scenes, results in accuracy_report.json). The final validated numbers are: 1.63 cm mean error, 100% exact count, 100% recall (185/185 cubes) at 2.0× cube spacing. No plots exist — no matplotlib histogram of per-scene errors, no error-vs-cube-count breakdown chart, no confusion matrix. Adding these to the clustering README and to the site would make the validation claim much more credible for the report. See clustering.html → Dataset.
  • Validate DBSCAN on real D435i hardware
    The entire clustering pipeline was validated on synthetic Gazebo data only. Real sensor data has different noise characteristics (quantisation, structured IR interference, specular reflections on cube faces). Needs a capture session with known cube positions. The coordinate transform and calibration parameters are designed for real deployment — just not yet confirmed.
  • Live centroid feed into RL policy
    Currently the pipeline is offline: PLY file → cluster → centroids. A live ROS 2 node that subscribes to /camera/points, runs DBSCAN, and publishes centroids as a ROS topic would enable closed-loop deployment. The Python 3.10/3.11 boundary is the main obstacle — needs a subprocess bridge or a ROS-compatible Open3D build.

RL training

  • Train Isaac Lab grab v5
    Reward version 5 is designed and committed but not yet trained. It returns to v0's proven scales and adds exactly one change: PhysX self-collision enabled. v0 achieved 97% success; v5 should match that without the arm-through-arm behaviour. Run: .\isaaclab.bat -p ...train.py --task Template-Holoassist-Grab-Direct-v5 --num_envs 4096 --max_iterations 2500 --headless
  • Isaac Lab reach — run to 1500+ iterations
    Reach task is at 60–70% success after 500 iterations with monotonically improving reward. Needs ~1000–1500 more iterations + entropy coefficient tuning to converge. Wall-clock: ~40–60 min additional on RTX 4070 Ti SUPER. See simulation.html → Training Runs.
  • MuJoCo grasp + transport sub-policies
    Pan and Extend sub-policies are trained. Grasp and Transport sub-policies are designed but not converged. The coordinator state machine is implemented; it needs working weights for all stages to demonstrate full PAN → EXTEND → GRASP → TRANSPORT → RELEASE. See training.html → MuJoCo.
  • Domain randomisation for sim-to-real
    Isaac Lab has built-in domain randomisation (mass, friction, lighting) but it has not been tuned. A DR sweep is needed before deploying a trained policy on the real UR3e — without it, the policy may be brittle to sensor noise and physical variation.

Integration & deployment

  • ONNX export validation (MuJoCo → Isaac Sim)
    The MuJoCo sub-policies export to ONNX (opset 17, float32) via export_onnx.py. The Isaac Sim side needs a matching ONNX inference node that loads these weights and drives joint targets. RSL-RL also supports ONNX export for the Isaac Lab policies — this should be set up before training converges, not after. See training.html → ONNX Export.
  • End-to-end demo: perception → RL → real robot
    No end-to-end run has been done. The full pipeline is: real D435i → ROS 2 point cloud → DBSCAN centroids → Isaac Lab policy (or ONNX sub-policies) → UR3e joint commands. Each subsystem works in isolation; the integration seams have not been exercised.
  • Teleop demonstration data (Nic)
    Nic collected teleoperation demonstrations via Quest 3. These can be used as behaviour-cloning seed data or for demonstration-augmented PPO to warm-start the grasp policy. The data exists; integration into the training pipeline has not been attempted.

Architecture

Architecture
