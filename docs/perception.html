Design Decision

Why we didn’t use AprilTag again

The base HoloAssist project used apriltag_ros — a third-party ROS package that does all the detection work for you. Using it again would mean our “perception pipeline” is just calling someone else’s library. For this project we wanted the perception to be our own implementation.

Instead we built a full point cloud processing pipeline in Python: statistical outlier removal, height cropping, density-based clustering, and centroid extraction. Every step is code we wrote and tuned. The only external dependency is Open3D for point cloud I/O — the clustering logic itself is our own DBSCAN implementation backed by scikit-learn.

Continuity with the base project: We kept the same calibration approach. The eye-to-hand calibration system Nic built for HoloAssist (Park solver, OpenCV calibrateHandEye) established the camera→robot transform with <2 mm accuracy. That same transform concept is used here to convert point clouds from camera frame to world frame before clustering. The geometry is the same — only the detection method changed.

Original System — Nic Sabatini

AprilTag perception & calibration (base HoloAssist)

The base HoloAssist perception system was built by Nic Sabatini and validated on the real UR3e robot. It provides the sim-to-real bridge that both projects share: the camera→robot coordinate transform.

How it worked

  1. RealSense D435i streams RGB + depth to apriltag_ros
  2. workspace_board_node detects 4 corner tags on a physical 700×500 mm board and runs SVD (Kabsch algorithm) to lock the workspace_frame TF — RMS residual <0.02 m
  3. cube_pose_node looks up AprilTag face tags (6 tags/cube, tag36h11 family) in workspace_frame and publishes cube poses at 20 Hz
  4. cube_pose_relay.py (Nic) bridges poses to base_link frame for the robot
  5. CubePoseSubscriber.cs (Nic) subscribes in Unity for XR hologram overlay
  6. A RANSAC depth plane fitting fallback in workspace_perception_node provided marker-free depth-based detection when tags were unavailable

Eye-to-hand calibration (Nic)

Nic built a full calibration node (eye_hand_calibration_node.py) using OpenCV’s calibrateHandEye with the Park solver. The node observes an AprilTag attached to the end effector from multiple robot poses and solves the static camera→base_link transform.

  • Sim mode: synthetic observations, validated Park solver at 0.4 mm accuracy
  • Hardware mode: real apriltag_ros detections captured via ROS services
  • Result: <2 mm reprojection error with 12–15 samples
  • Saved to ~/.ros2/easy_handeye2/calibrations/ — survives reboots

New System — Phase 1

Point cloud capture (ROS 2 + Gazebo)

The simulated Intel RealSense D435i in Gazebo Ignition Fortress publishes RGB-D point clouds on /camera/points (PointCloud2). A ROS 2 node saves them as binary PLY files for offline clustering.

Key transform insight

Point clouds are published in the camera body frame (+X = look direction). Any height-based cropping must happen after transforming to world frame — otherwise the crop plane is tilted with the camera. This is the same principle as Nic’s cube_pose_relay.py, which bridged poses through TF to get them into base_link frame.

New System — Phase 2 — Guy Smith

DBSCAN clustering pipeline

Guy Smith built and validated the full DBSCAN pipeline, replacing an initial K-Means baseline. The key advantage over both K-Means and AprilTag: no prior knowledge of cube count required, and no physical markers.

Evaluation

Guy built a dataset capture pipeline (dataset_capture.py) that automatically captures 60 scenes (50 train + 10 val) with 2–4 cubes per scene, saving PLY files plus ground-truth cube poses read directly from Gazebo physics state. A Hungarian matching benchmark (verify_detection.py) measures centroid accuracy against ground truth.

Key fix: cube spacing

The 82% exact-count rate was caused by adjacent cubes merging into a single cluster when placed at 1.5× cube size apart (2 cm surface gap). DBSCAN with eps=0.015 m bridges the gap when point cloud noise fills it. Guy fixed the minimum cube spacing to 2.0× (4 cm surface gap), which reliably separates clusters. The dataset is being regenerated to validate this.

Why XYZ-only clustering

Early experiments with RGB colour weighting in the clustering confused DBSCAN under Gazebo’s lighting: shadows and specular highlights caused the same cube to produce points with wildly different colour values, splitting single clusters into fragments. Setting color_weight = 0.0 (XYZ only) resolved this entirely.

Evaluation

Results & limitations

Successes

  • 1.63 cm centroid accuracy is within 3.3× sensor noise (σ=0.005 m) — close to the theoretical floor for this sensor
  • 100% cube recall across all validated scenes — every cube detected in every scene
  • No physical markers required, works on any distinctly coloured object in the workspace
  • Calibration continuity: same camera→robot transform principle as base HoloAssist, ensuring sim-to-real validity

Limitations

  • Cube merging at close range: DBSCAN merges cubes placed <2× cube size apart. Fixed by enforcing wider spacing — validated data pending.
  • Simulation only: DBSCAN pipeline validated in Gazebo sim, not yet tested on real RealSense hardware. The transform chain and calibration parameters are designed for real deployment but not yet confirmed.
  • Fixed height crop: pipeline assumes known table height and cube dimensions. Dynamic scenes with varying object heights would require plane-fitting (e.g. RANSAC — as used in the original workspace_perception_node).
  • Colour-agnostic: XYZ-only clustering cannot distinguish objects of different types at the same height. For mixed scenes, colour filtering would need to be re-introduced carefully.

Future work

  • Validate DBSCAN at 2.0× cube spacing — expected to push exact-count rate above 90%
  • Test on real hardware using the saved easy_handeye2 calibration from base HoloAssist
  • Feed live centroid stream directly to Isaac Sim policy (currently offline PLY → cluster)
