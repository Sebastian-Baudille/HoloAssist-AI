Overview

Why Isaac Sim for final training

Isaac Sim (Omniverse-based) enables GPU-parallelised physics at 4096+ simultaneous environments — the scale needed for PPO to converge in reasonable wall-clock time on hard manipulation tasks. MuJoCo (John’s component) runs efficiently on CPU with 16 envs; Isaac Sim at 4096 envs achieves similar sample throughput per hour at 256× the episode count per batch.

Strategy-module design

Both the reach and grab tasks follow a strategy-module architecture: the env class is a thin shell that handles lifecycle (init, scene setup, reset, dones), and all observation, reward, and action logic lives in sibling subpackages:

tasks/direct/reach/ observations/ground_truth_12d.py ← observation strategy rewards/dense_reach_v1.py ← reward strategy (active) actions/joint_delta.py ← action strategy reach_env.py ← env shell (imports strategies at top)

To A/B test a different reward design: change one import line, no env subclassing needed. Multiple reward versions are kept simultaneously (dense_reach_v0 through dense_reach_v3_ik). The grab task uses a dispatch dict so reward versions can be selected by config string rather than requiring separate env registrations.

Robot Asset

URDF pipeline & gripper mimic problem

The UR3e + RG2 URDF required preprocessing before Isaac Sim could load it:

  1. inspect_urdf.py — validates joint types, link masses, mesh paths
  2. prepare_urdf.py — converts fixed joints to fused links where needed; sets up RG2 gripper’s mimic joints in a form Isaac Sim can handle
  3. Isaac Sim URDF importer → layered USD asset
ur_onrobot.urdf → Isaac Sim URDF importer → ur_onrobot_prepared.urdf (physics preparation) → ur_onrobot_prepared_base.usd + physics.usd + robot.usd + sensor.usd → ur_onrobot_prepared.usd (composed asset)

PhysxMimicJointAPI problem

The RG2 gripper is a 4-bar parallel linkage. In URDF the follower joints are defined as <mimic> joints. Isaac Sim 5.1’s PhysxMimicJointAPI is metadata-only — it does not enforce mimic behaviour in the physics solver. This was proven empirically during development.

Solution: Enforce gripper kinematics via stiff position drives. All 6 linkage joints (master + 5 followers) are given commanded positions computed with the correct gear signs (common/gripper_coupling.py), and drive stiffness is set high enough (linkage_drive_stiffness, typically 1000–5000) that they track the command rigidly under contact.

# In grab_env.__init__: stiff = torch.full((num_envs, 6), cfg.linkage_drive_stiffness, device=device) robot.write_joint_stiffness_to_sim(stiff, joint_ids=linkage_ids_tensor)

EE proxy

Isaac Sim’s “merge fixed joints” pass fuses frame-only links with no inertia into their parent. gripper_tcp (the true TCP frame) gets merged into onrobot_base_link and is not queryable. The closest available body approximating the grasp point is left_inner_finger. For the grab task, gripper centre is the midpoint of both inner fingers:

gripper_center = (left_finger_pos + right_finger_pos) * 0.5

Task 1

Reach task

Drive the end-effector (left_inner_finger body) to a randomised XYZ target position above the table. No gripper control — arm only. Success when EE is within success_tolerance_m of the target.

Observation space (12D — ground_truth_12d.py)

[ee_x, ee_y, ee_z, end-effector world position (3) target_x, target_y, target_z, target world position (3) j0, j1, j2, j3, j4, j5] arm joint positions (6)

Action space (6D — joint_delta.py)

action[i] ∈ [-1, 1] → joint_i_delta = action[i] × cfg.action_scale_rad # Applied as position targets: joint_pos_target += delta

IK grid (v3 only)

The reach env precomputes a 20×20 IK grid over the target spawn zone at init time (~2 s startup cost, amortised over the full run). At each reset, the nearest-neighbour IK solution is looked up and stored as _ik_reference for the tracking reward term.

Task 2

Grab task — reward design iterations

Reach a cube, close the gripper around it, and lift it at least 5 cm above the table surface. This is the most technically involved part of the Isaac Sim stack.

Observation space (16D)

[ee_x, ee_y, ee_z, gripper centre world position (3) cube_x, cube_y, cube_z, cube world position (3) j0–j5, arm joint positions (6) gripper_close_signal, action[6] from previous step (1) finger_width] actual jaw width from physics (1) + 2 additional terms per version

Action space (7D)

action[0:6] ∈ [-1, 1] → arm joint deltas × cfg.action_scale_rad action[6] ∈ [-1, 1] → gripper signal (+1 = open, -1 = close)

Version 0 — working baseline

6 reward terms (proximity-gated alignment, grasp activation, lift bonus, success bonus). Result: 97% success, 196/200 mean reward. Behaviour: “side-sprawl” — arm approaches from the side rather than above, with visible self-collision. Self-collision physics was disabled in v0.

Version 1 — hover trap

Added approach_height bonus and made orientation reward ungated. Result: arm learned perfect overhead posture, then hovered indefinitely without closing the gripper. Mean episode length 200 (always timeout). The policy earned ~1.5/step for maintaining orientation without ever attempting to grasp.

Design lesson: When orientation or alignment rewards are not gated on proximity, the policy will find a pose that maximises them continuously rather than moving toward grasping.

Version 2 — lateral misalignment

Added time_penalty (-1/step) to break the hover. Rebalanced scales. Result: two new exploits — (1) hold-below-threshold: gripper beside the cube, cube raised just below success threshold, milking continuous lift bonus without triggering termination; (2) sloppy grasp: arm closed gripper anywhere near the cube for grasp-activation reward.

Design rule established: sum(per-step reward) × max_episode_length MUST be < success_bonus. When this invariant breaks, the policy prefers to milk per-step rewards over terminating with success. v1 and v2 both violated this.

Version 3 — finger drag

Strategic retreat to v0’s 6-term design + PhysX self-collision + raised success threshold (10 cm) + small elbow_up nudge. Result: approach posture was correct — arm overhead, wrist-down, centred. But fingers descended to the table surface with the cube between them and the gripper did not close. Once fingers touch the table, friction prevents inward closing motion. The policy earned ~190/episode from alignment rewards while stuck at the cube.

Version 4 — safe hover

Added anti_drag term: -1/step when finger_z ≤ table_top + 0.20. Result: policy learned to hover at exactly 0.21 m (just above threshold) indefinitely. Mean reward: ~13 — the worst result of all versions. The anti-drag pushed the policy away from the table but provided no gradient toward grasping.

Version 5 — conservative return (pending)

Return to v0’s exact reward scales (the only configuration empirically proven to work). Add only: PhysX self-collision (fixes v0’s arm-through-arm flaw) + tiny elbow_up nudge. No scale rebalancing, no new exploration constraints. Drops the 10 cm aspiration — 5 cm is the threshold proven achievable with the v0 reward structure. Not yet trained at time of writing.

Discussion

Reflections on the reward shaping challenge

The five reward iterations for the grab task illustrate the central challenge in dense reward manipulation RL: the policy will find and exploit any loophole in the reward function. Each version closed one loophole but opened another:

  • v0: reward achievable via self-collision (close proximity + no physics constraint)
  • v1: reward achievable via hovering indefinitely (ungated orientation bonus)
  • v2: reward achievable via holding cube below threshold + random closing
  • v3: reward achievable via finger-drag (approach correctly, never close)
  • v4: reward achievable via safe hover (stay above threshold, never descend)

v5’s design principle: return to the only configuration empirically proven to work (v0) and change exactly one thing (enable self-collision). This conservative iteration discipline — make one change at a time and verify each change doesn’t introduce a new exploit — is slow but reliable.

What we would change

  • Self-collision from the start: v0’s working policy has visible arm self-collision. Enabling PhysX self-collision from the very first grab run and designing the reward to be compatible with it would have saved several failed reward iterations.
  • Invariant check before training: The “sum of per-step rewards × episode_length < success_bonus” invariant was discovered empirically. This should be verified analytically before any training run starts.
  • Curriculum for closing: v3 and v4 both show the policy can reach the correct position but cannot learn to close the gripper. A curriculum rewarding the gripper-close action specifically when correctly positioned might provide the gradient needed to break out of these traps.
