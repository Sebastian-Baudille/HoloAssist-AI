Overview

What the Gazebo stack does

The Gazebo simulation is the data source for the project — not the RL training environment. Its four responsibilities:

  1. Render a physics-accurate scene with a UR3e + RG2 arm, a table, and colour-coded 4 cm cubes
  2. Publish a synthetic RGBD point cloud over ROS 2, mimicking an Intel D435i camera
  3. Provide a scene controller that randomises cube positions on demand
  4. Capture labelled PLY files as training data for the DBSCAN clustering pipeline

The PLY files written to ~/holoassist_pointclouds/ are the bridge to the Python 3.11 clustering environment. RL training itself runs in MuJoCo (John) and Isaac Sim (Seb).

Architecture

Template-driven world generation

Rather than hardcoding a fixed SDF world file, the sim uses a Jinja2 template pipeline:

sim_params.yaml (single source of truth) ↓ render_world.py (Jinja2 renderer) ↓ table_cubes.sdf (rendered into /tmp at launch time) ↓ gz sim -r table_cubes.sdf

Changing the camera position, number of cubes, table size, or any physical property is a YAML edit — no SDF editing required. The same pattern applies to the ROS–Gazebo bridge config (ros_gz_bridge.yaml.jinja2).

sim_params.yaml — single source of truth

All scene and camera parameters live in ros2_ws/src/holoassist_sim/config/sim_params.yaml:

KeyPurpose
table.{size,pose,color}Table solid block geometry
cubes[].{name,size,pose,color,mass}Free list — add/remove cubes without touching SDF
camera.{name,pose,topic,...}D435i parameters (pose, FOV, resolution)
camera.imu.{enabled,update_rate,topic}IMU sensor toggle
capture.{output_dir,description,one_shot,voxel_size}PLY capture settings

The default camera pose is:

pose: [-0.6, 0.0, 0.85, 0.0, 0.5281, 0.0] # x y z roll pitch yaw

This places the camera 0.6 m in front of the robot, 0.85 m up, tilted 30° downward (0.528 rad pitch) — a wide-angle view of the table surface.

Launch pipeline

sim.launch.py orchestrates everything in order:

  1. Read sim_params.yaml
  2. Render the SDF and bridge config into a temp directory
  3. Spawn gz sim -r <rendered.sdf> (Ignition Fortress)
  4. Spawn ros_gz_bridge with the rendered config
  5. Publish two static TFs: map → d435i_link → d435i/d435i_link/rgbd
  6. Spawn RViz2 with the bundled rviz config
  7. Spawn scene_controller.py for cube randomisation

TF chain

map → d435i_link → d435i/d435i_link/rgbd

Two transforms are required because Ignition Fortress scopes the sensor frame as <model>/<link>/<sensor_name>. The published frame_id is d435i/d435i_link/rgbd, not a simple camera_link. An early version only published one TF; the point cloud had no chain to map and the entire clustering pipeline produced wrong results.

Frame convention: Camera body frame is +X = look direction, +Z = up. This is the opposite of the standard ROS camera optical frame (+Z forward, -Y up). Any height-based Z-crop must happen after transforming to world frame — otherwise the crop plane tilts with the camera.

Scene controller

scene_controller.py is a ROS 2 node that manages cube spawning at runtime:

  • Exposes /scene/randomize_cubes (Trigger service) — spawns N cubes at random positions
  • Exposes /scene/reset — removes all random cubes
  • Parameters (cube_count, cube_size_min, cube_size_max) editable live via rqt_reconfigure
  • Overlap prevention: rejection-sampled with minimum separation = 2.0× cube size (was 1.5×, see Problems)

PLY Capture

save_pointcloud.py

A ROS 2 node that subscribes to /camera/points (PointCloud2) and writes one PLY file per capture.

Key implementation details

  • BEST_EFFORT QoS: Ignition Fortress publishes sensor data with best-effort reliability. A reliable subscriber advertises a different QoS profile and the bridge silently drops all messages. Using BEST_EFFORT fixed this.
  • RGB unpacking: RGB is packed as a float32 bit-cast from uint32 0x00RRGGBB. The node unpacks it to uint8 R, G, B.
  • Binary PLY: Written as binary little-endian. Open3D and MuJoCo both expect binary PLY; ASCII causes decoder failures downstream.
  • Optional voxel downsampling: Hash-based uniqueness filter when voxel_size > 0.

Naming convention

<description>_<N>cubes_<size>mm_v<NNN>.ply # auto-versioned # Example: default_4cubes_40mm_v001.ply

Output goes to ~/holoassist_pointclouds/. Version auto-increments so successive captures don’t overwrite.

Dataset Capture

dataset_capture.py — automated dataset generation

A more advanced ROS 2 node that orchestrates 60 scenes (50 train + 10 val) automatically:

01

Set Cube Count

Sets cube_count parameter on scene_controller via direct service call.

›
02

Randomise + Settle

Calls /scene/randomize_cubes, then waits 1 s for physics to settle.

›
03

Read Settled Poses

Reads actual settled positions from Gazebo via ign topic protobuf output. Not commanded poses — actual physics state.

›
04

Save PLY + Labels

Captures one PointCloud2 frame, writes scene_NNNN.ply + scene_NNNN.labels.json with ground truth.

Labels JSON format

{ "scene_id": "scene_0001", "split": "train", "cube_count": 3, "table_top_z": 0.5, "cubes": [ {"name": "cube_rand_00", "position": {"x": 0.12, "y": -0.08, "z": 0.52}, "orientation": {"x": 0, "y": 0, "z": 0, "w": 1}, "size_m": 0.04} ] }

Cube size is back-calculated from settled Z position: size = (cube_z − table_top_z) × 2. Reading settled poses rather than commanded poses correctly captures cubes that have drifted or stacked due to physics, giving the clustering pipeline honest ground truth.

# Capture a full labelled dataset (60 scenes) ros2 run holoassist_sim dataset_capture \ --params ros2_ws/src/holoassist_sim/config/sim_params.yaml

Dataset structure

~/holoassist_dataset/ scene_0001.ply scene_0001.labels.json ... scene_0060.ply (50 train + 10 val) accuracy_report.json

Problems Encountered

Issues & fixes during development

1 — Depth noise and Z-crop margin

The camera has Gaussian depth noise σ=0.005 m. The initial clustering Z-crop cut exactly at table_top_z, causing table surface noise to bleed into the cube layer. The fix was a 3× noise margin:

z_min = table_top_z + 0.015 # 3 × σ

2 — BEST_EFFORT QoS requirement

When save_pointcloud.py used a reliable QoS subscriber for /camera/points, no messages were received. Ignition Fortress publishes sensor data with BEST_EFFORT reliability — a reliable subscriber advertises a different QoS profile and the bridge silently drops all matching attempts. Changing the subscriber to BEST_EFFORT fixed this immediately.

3 — Cube separation: 1.5× → 2.0×

The initial scene controller used 1.5× cube size as the minimum separation between cube centres (~2 cm surface gap for 4 cm cubes). DBSCAN with eps=0.015 bridges this gap when depth noise fills it, merging two adjacent cubes into one cluster. The separation was increased to 2.0× (~4 cm surface gap) to ensure adjacent cubes stayed outside each other’s neighbourhood radius.

4 — Frame convention confusion

The point cloud is published in camera body frame (+X forward, +Z up), not the standard ROS camera optical frame (+Z forward, -Y up). Early versions of the clustering pipeline tried to crop by world Z without first transforming from camera frame to world frame, producing completely wrong results.

# Correct pipeline order: world_points = camera_to_world(camera_frame_points, camera_pose) # THEN do Z-crop cropped = world_points[world_points[:, 2] > table_top_z + 0.015]

5 — Two static TFs required

An early version only published one static TF (map → camera_link). The point cloud’s frame_id was d435i/d435i_link/rgbd (the Ignition-scoped name), which had no TF chain to map. RViz showed no point cloud, and transformation in Python had to be done manually. Adding a second TF (camera_link → d435i/d435i_link/rgbd, identity) fixed both RViz and the clustering pipeline.

Discussion

Design decisions & reflections

Why Ignition Fortress?

ROS 2 Humble’s officially supported simulator is Ignition Fortress (ign-gazebo6), which replaced Gazebo Classic. The bridge (ros_gz_bridge) translates between Ignition’s internal transport and ROS 2 topics. This added complexity (the two TF setup, BEST_EFFORT QoS quirks, scoped frame_id) compared to Gazebo Classic, but Fortress is the supported long-term choice for Humble.

Why PLY as the bridge format?

The clustering pipeline runs in Python 3.11 (for Open3D and Polyscope compatibility) while ROS 2 Humble requires Python 3.10. Rather than mixing environments or calling across processes, PLY files on disk serve as a clean, version-agnostic bridge. PLY is a binary format natively supported by Open3D, Polyscope, and NumPy, and carries both XYZ and RGB without additional libraries. It also creates a natural checkpoint — captures can be reviewed and re-analysed without re-running the simulation.

Why not use Gazebo for RL training?

Earlier in the project there was an attempt to use Gazebo Classic with a PPO training loop, with keyboard teleoperation and MoveIt safety layer. This was abandoned because Gazebo Classic is too slow for RL training (cannot run >1 environment in parallel without significant infrastructure) and MoveIt overhead adds latency that makes tight control loops impractical. The RL training was moved to MuJoCo (John) and Isaac Sim (Seb), while Gazebo became the perception data source.

What we would change

  • Decouple sensor frame from Ignition naming: The frame_id embedded in published PointCloud2 messages is tied to Ignition’s internal naming scheme. A wrapper node republishing with a fixed frame_id like camera_optical would be more robust.
  • Larger dataset: 60 scenes is small. With dataset_capture.py the bottleneck is the 1 s settle time per scene, so scaling to 500+ scenes would take ~8 minutes — completely feasible.
  • Cube variation: All cubes have fixed 4 cm size. Adding size variation (cube_size_min/cube_size_max already supported) and random yaw would make the pipeline more robust.
