Environment Setup

Python environment constraint

The clustering pipeline has a hard Python environment constraint. Open3D and Polyscope require Python 3.11, but rclpy (ROS 2 Python client) only runs under Python 3.10 (Humble). The two environments cannot coexist in the same process.

Solution: PLY files on disk as the cross-environment bridge. No ROS topics or rclpy imports appear anywhere in the clustering code. No Open3D or Polyscope imports appear anywhere in the ROS nodes.

Approach

K-Means to DBSCAN — why we switched

The project went through three approaches to cube detection:

MethodErrorExact-countLimitation
AprilTag 3 (base HoloAssist)~5–10 mm100%Physical markers on every cube face. Not our code.
K-Means (initial baseline)2.65 cm~90%Requires knowing cube count k upfront. Assigns every point (including noise) to a cluster.
DBSCAN, 1.5× spacing (early)1.63 cm82%Adjacent cubes merged at eps=1.5 cm gap
DBSCAN, 2.0× spacing (validated)1.63 cm100%185/185 cubes, zero missed or merged

DBSCAN wins on two metrics that matter: better accuracy than K-Means (because it labels noise as −1 and discards it rather than forcing it into a cluster), and no need to know how many cubes are in the scene.

Why no RGB weighting?

Both K-Means and DBSCAN were tested with RGB colour weighting. Gazebo’s diffuse lighting model creates significant per-pixel variation — the same cube colour appears very different on a lit face vs a shadowed face. Including RGB weighting caused clusters to split along lighting boundaries rather than grouping by object. Setting color_weight = 0.0 (XYZ-only clustering) resolved this entirely.

Stage 1

K-Means baseline

The first approach was K-Means clustering with a fixed k. The motivation was simplicity — K-Means is deterministic, fast, and widely understood. If you know there are 4 cubes, k=4 directly gives 4 centroids.

Camera-to-world transform

The point cloud is in camera body frame (+X = look direction). Before any Z-based cropping can work, all points must be transformed to world frame using the camera pose from sim_params.yaml ([x, y, z, roll, pitch, yaw]):

R = Rz @ Ry @ Rx world_points = (R @ camera_points.T).T + [x, y, z] # THEN do Z-crop

This was a critical early correctness requirement — forgetting to transform first meant the Z-crop was applied in the wrong frame, selecting nothing or the wrong layer.

Z crop parameters

z_min = table_top_z + 0.015 # 3× sensor noise σ (0.005 m) z_max = table_top_z + cube_height + 0.01

K-Means results

First validated accuracy: ~2.65 cm mean centroid error. This passed the 3 cm target, but K-Means has a fundamental limitation: k must be specified, and it forces exactly k clusters even when cubes are occluded, merged, or the crop contains noise.

Validation

Dataset & evaluation methodology

Guy built a fully automated dataset capture pipeline (dataset_capture.py) that runs inside the Gazebo simulation and captures 60 scenes with 2–4 cubes per scene at random positions. Ground-truth cube poses are read directly from Gazebo’s physics state at capture time — not estimated, not commanded positions.

Hungarian algorithm matching

verify_detection.py uses the Hungarian algorithm (scipy.optimize.linear_sum_assignment) to optimally pair detected centroids to ground-truth cube positions. Because DBSCAN may detect more or fewer cubes than ground truth:

  • Missed cubes (detected < GT): unmatched GT cubes get a penalty distance of 0.50 m so misses register as large errors in the mean
  • Extra detections (detected > GT): counted as false positives but not added to the error metric

This produces honest accuracy numbers — a pipeline that misses half the cubes doesn’t hide behind averaging only the cubes it found.

The spacing bug & fix

Early in development the scene controller used 1.5× cube size as minimum separation (~2 cm surface gap), giving an 82% exact-count rate and ~91% recall because DBSCAN with eps=0.015 m would bridge the narrow gap between close cubes. Minimum separation was increased to 2.0× (~4 cm surface gap), the dataset was regenerated, and the validated results are the 100%/100% figures above.

Results

Results & limitations

Successes

  • 1.63 cm centroid accuracy is within 3.3× sensor noise floor (σ=0.005 m) — strong given that depth noise is the fundamental limit
  • 100% cube recall: every cube detected in every validated scene
  • No physical markers or prior cube count needed — generalises to any number of objects in the height band
  • XYZ-only clustering is robust to Gazebo lighting variation where RGB-weighted clustering failed
  • The 1.63 cm accuracy is well within the robot’s end-effector jaw margin — perception error is not the RL training bottleneck

Limitations

  • Close-range merging: cubes <2× cube size apart merge into one cluster. Dataset spacing fix in progress.
  • Fixed geometry: height crop assumes known table height and cube size. A dynamic scene would need plane-fitting (e.g. RANSAC — as used in the original workspace_perception_node).
  • Simulation only: validated in Gazebo, not yet confirmed on the real D435i sensor.
  • Colour-agnostic: XYZ-only clustering cannot distinguish objects of different types at the same height.

Impact on RL training

The centroid output (world-frame XYZ, shape (12,) for 4 cubes) feeds directly into the PPO observation vector. At the 95th percentile (worst cubes), error rises to ~3–5 cm — at the edge of the jaw margin but still within the gripper’s geometric range of acceptance.

Discussion

Design decisions & reflections

K-Means vs DBSCAN

Both algorithms were tried, and DBSCAN won on two dimensions: accuracy and flexibility. The accuracy improvement (2.65 cm → 1.63 cm) came primarily from better noise handling — DBSCAN labels noise explicitly as −1 and discards it before computing centroids, whereas K-Means is forced to assign every point (including noise) to a cluster, polluting centroids.

The main cost of DBSCAN is that eps and min_samples must be tuned. At 1.5 cm eps, clusters 1.5 cm apart merge; at 2.5 cm they reliably stay separate but noise rejection worsens. The current 1.5 cm value is a compromise that may need revisiting if the table scene changes.

Why not deep learning?

A learned detector (PointNet, PointPillars, or a simple voxel CNN) would likely outperform DBSCAN for occluded cubes and variable lighting. The decision to use classical clustering was driven by:

  1. Time: the Gazebo + clustering stack needed to be functional quickly to unblock the RL pipeline
  2. Interpretability: classical clustering failures are visually diagnosable in Polyscope; a learned model’s failures are less obvious
  3. Data: 60 scenes is insufficient to train a reliable 3D object detector from scratch

What we would change

  • Adaptive eps: current eps=0.015 is tuned for 4 cm cubes at ~0.7 m range. Cubes closer produce denser point clouds; farther ones produce sparser clouds. Adaptive eps scaling with estimated range would be more robust.
  • Colour-aware DBSCAN: HSV hue is more lighting-invariant than RGB. Using hue to separate cubes that are very close spatially could improve the close-cube merging problem.
  • Larger dataset: 60 scenes is functional but small. dataset_capture.py can easily scale to 500+ scenes in under 10 minutes.
  • 3D bounding box output: the PCA axes and extents could produce an oriented bounding box per cube, useful for grasp orientation planning.
  • Real-camera validation: the entire pipeline was validated on synthetic Gazebo data. Real D435i data has different noise characteristics (quantisation, structured IR interference, specular reflections).
